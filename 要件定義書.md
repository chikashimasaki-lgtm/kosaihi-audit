## 1. 背景・目的

### 1.1 背景
企業の経費精算において、会議費・交際費・福利厚生費・旅費交通費などの勘定科目は、税務上の取扱い（特に交際費の損金不算入）に直結する。一方で、現場では以下のような不適切処理・統制上のリスクが発生しやすい。

- 交際費を会議費等へ付け替える「隠れ交際費」
- 1人あたり1万円基準の超過判定漏れ
- インボイス未登録店に対する経過措置（仕入税額控除の制限）を踏まえない金額判定
- 事前決裁を経ない事後精算・決裁遅延
- 高級店利用・役職不整合による「相手方偽装」「過剰接待」リスク

内部監査部門がこれらを全件目視チェックするのは非効率であり、**機械的に判定可能な部分を自動化し、疑義のある案件のみを抽出**する仕組みが求められる。

### 1.2 目的
経費精算データを入力するだけで、ルールベースの自動判定により以下を実現する。

1. 税務・会計・決裁プロセス上の疑義を自動検知し、4段階で格付けする
2. **疑義案件のみ**「監査内容確認書」を自動生成し、被監査部門への質問形式で出力する
3. 監査結果・確認書を CSV / Excel で出力し、監査調書として利用可能にする
4. 定性判定が必要なケースに向け、外部生成AI（Copilot / Gemini 等）用プロンプトを提供する

### 1.3 スコープ外（非目的）
- 本ツールは**一次スクリーニング専用**であり、最終的な税務処理・損金判定を確定するものではない
- 隠蔽・偽装の検知は「疑義の示唆」であり、不正の断定はしない
- 会計システム・ワークフローシステムとの自動連携は行わない（手動でのデータ貼り付け方式）

## 2. システム全体方針

### 2.1 基本コンセプト
| 区分 | 方針 |
|------|------|
| 動作環境 | 単一HTMLファイル（ブラウザ完結型 / 静的サイト） |
| 処理方式 | **端末内のルールベース計算（数式）のみ**。AI・外部APIは一切使用しない |
| データ保護 | 入力データは外部送信しない（ローカル処理のみ） |
| 提供形態 | GitHub Pages による静的ホスティング |

### 2.2 判定の3ステップ構成
1. **STEP1 機械的チェック**：税抜計算・1万円基準・決裁日付（確定値）
2. **STEP2 不祥事シナリオ**：高級店×役職不整合、事前申請回避などの隠蔽リスク（疑義の示唆）
3. **STEP3 定性判定**：事後決裁理由の妥当性等。外部生成AI用プロンプトを提供（手動運用）

## 3. 利用者・利用シーン

### 3.1 想定利用者
| 利用者 | 役割 |
|--------|------|
| 内部監査部門 | 経費精算データを投入し、疑義案件を抽出。確認書を発行する |
| 経理部門 | 月次・期末の経費チェックの一次スクリーニング |
| 被監査部門 | 発行された監査内容確認書に回答・証跡提示を行う |

### 3.2 利用シーン
- 月次の経費精算データに対する定期スクリーニング
- 期末の交際費集計前のチェック
- 内部統制（J-SOX等）対応のための証跡作成

### 4.2 入力データ項目（14列）
列順固定。タブ区切りを優先し、タブがなければカンマ区切りとして解釈する。

| No | 列名 | 用途 |
|----|------|------|
| 1 | 精算申請ID | 案件の一意識別 |
| 2 | 関連決裁番号 | 空欄＝事前申請なしと判定 |
| 3 | 支出日 | 決裁プロセス判定の基準日 |
| 4 | 事前承認日 | 支出日より後なら事後決裁と判定 |
| 5 | 精算決裁日 | 支出日との差が猶予超なら決裁遅延 |
| 6 | 金額税込 | 税抜計算の元金額 |
| 7 | 人数 | 1人あたり金額の算出 |
| 8 | 社外人数 | 0なら社内飲食、>0なら接待系判定 |
| 9 | インボイス | 「未」を含むと未登録店扱い（経過措置適用） |
| 10 | 店舗 | 費目カテゴリ・高級店判定 |
| 11 | 申請科目 | 申請時の勘定科目（判定科目との突合対象） |
| 12 | 社内最上位役職 | 役職不整合判定 |
| 13 | 社外最上位役職 | 役職不整合判定 |
| 14 | 事後決裁理由 | 定性判定の材料 |

### 4.3 判定要件

#### 4.3.1 税抜計算（監査用税抜価額）
- 経理方式は**税抜経理固定**
- 本体価格 = 金額税込 ÷ (1 + 税率)、消費税額 = 金額税込 − 本体価格
- **登録店**：監査用税抜価額 = 本体価格
- **未登録店**：監査用税抜価額 = 本体価格 + 消費税額 × 控除不可率（2026年=20%）
- 1人あたり = 監査用税抜価額 ÷ 人数

#### 4.3.2 費目カテゴリ判定（店舗名・申請科目から推定）
| カテゴリ | 判定条件 |
|----------|----------|
| ゴルフ | 店舗名に「ゴルフ／カントリー／CC」等を含む |
| 手土産 | 店舗名に「手土産／菓子折／贈答／百貨店／中元／歳暮」等を含む |
| 旅費 | 申請科目が旅費交通費、または店舗名に「出張／宿泊／ホテル／新幹線」等を含む |
| 飲食 | 上記以外 |

#### 4.3.3 勘定科目の判定ルール
| カテゴリ | 判定 |
|----------|------|
| ゴルフ・手土産 | 社外あり→交際費、社内のみ→福利厚生費申請なら福利厚生費・それ以外は交際費。申請科目と不一致なら「隠れ交際費」フラグ |
| 旅費 | 社外同伴→交際費の疑い、社内のみ→旅費交通費 |
| 飲食（社内のみ） | 福利厚生費申請＝適正、それ以外＝交際費（社内飲食費） |
| 飲食（社外あり） | 1人あたりが基準超→交際費、基準以下→会議費。福利厚生費に社外混入→実態交際費 |

#### 4.3.4 決裁プロセス判定
- 関連決裁番号が空 → 「事前申請なし」
- 事前承認日 > 支出日 → 「事後決裁（理由あり／なし）」
- 精算決裁日 − 支出日 > 猶予日数 → 「決裁遅延（n日）」

#### 4.3.5 不祥事シナリオ（隠蔽・偽装リスク）
- 高級店（割烹／料亭／会員制／鮨／銀座／鉄板 等）× 社内上位役職 × 社外下位役職 → 「相手方偽装疑い」
- （高級店 または ゴルフ）×（事前申請なし／事後決裁）→ 「隠蔽目的の事前申請回避」

#### 4.3.6 格付け（4段階）
| 格付け | 条件 |
|--------|------|
| 🟥 要改善 | 相手方偽装疑い、または隠蔽目的の事前申請回避 |
| 🟨 要改善(軽微) | 隠れ交際費／社内飲食費／福利厚生費混入／決裁遅延／事前申請なし／事後決裁(理由なし)／接待旅費 |
| 🟦 注意 | 事後決裁(理由あり) |
| 🟩 適正 | 上記いずれにも該当しない |

### 4.4 監査内容確認書の生成要件
- 疑義案件（適正以外）のみ生成する。適正案件は自動スルー
- 検知フラグに応じて被監査部門への質問（Q1, Q2…）を動的に生成する
- 各質問に「ご回答：____」記入欄を設ける
- 構成：1.対象案件 / 2.監査人の所見（税務計算・決裁プロセス・隠蔽リスク）/ 3.質問 / 4.提示証跡チェックリスト / 回答期限・回答者欄
- 是正の一方的指示ではなく、被監査部門が回答・説明する**質問形式**とする

## 5. 非機能要件

| 区分 | 要件 |
|------|------|
| 性能 | 数百件規模のデータを即時（数秒以内）に判定 |
| セキュリティ | 入力データを外部送信しない。サーバ側に保持しない |
| 可用性 | 静的サイトのため依存サービス障害の影響を受けにくい |
| 保守性 | 単一HTMLファイル完結。判定パラメータは画面上で変更可能 |
| 互換性 | モダンブラウザ（Chrome/Edge/Safari）。日本語フォント対応 |
| 出力互換 | CSVはBOM付きUTF-8（Excel文字化け防止）、Excelは.xls（HTMLテーブル方式） |

## 6. 制約・前提条件

- 本ツールは**デモアプリ**であり、サンプルデータでの動作確認を主目的とする
- 機械判定（税抜計算・日付・1万円基準）は確定値、隠蔽／偽装は「疑義の示唆」である
- 最終判定・税務処理は専門家・原資料で確認する必要がある
- 経理方式は税抜経理固定、経過措置の控除不可率は2026年=20%を初期値とする
- インボイス登録／未登録は入力データの「インボイス」列に依存する

## 7. リスク・留意事項

| リスク | 対応 |
|--------|------|
| 誤検知・検知漏れ | 一次スクリーニングと位置づけ、最終判断は人が行う |
| 店舗名ベースの推定限界 | 店舗名から費目・高級店を推定するため取りこぼしの可能性あり |
| 税制改正への追従 | 控除不可率・税率・基準額を画面上で変更可能とすることで対応 |
| 不正断定の誤り | 「疑義の示唆」に留め、質問形式での確認とする |
